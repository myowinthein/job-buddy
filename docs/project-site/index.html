Autofill job application forms from one saved profile

A Chrome extension that fills repetitive job application fields across thousands of employer and Applicant Tracking System sites, using a structured profile the user maintains once.

Job Buddy reads form fields on the page, matches them against a stored profile, and fills in what it can with per-field confidence indicators. Everything stays in the browser's local storage by default. Three optional features (Google Drive backup, AI resume import, AI autofill assist) call out to Google's APIs only when the user explicitly enables them.

Problem

Why this exists

Job seekers, especially those applying across multiple countries, retype the same personal details, work history, salary expectations, and work authorization information into dozens of slightly different forms. Every Applicant Tracking System (Workday, Greenhouse, Lever, and various in-house custom forms) has its own field naming, layout, and quirks, so generic browser autofill is too brittle to help reliably.

Job Buddy keeps one structured profile per user and applies it intelligently across any employer site, with per-field confidence so the user knows where to look before submitting.

Target users

Who this is for

The repository indicates the extension is designed for:

  • Active job seekers applying to multiple roles across different companies
  • Multi-country applicants who deal with varying work authorization, salary currencies, and address formats
  • Users who want their data to stay on their own device and prefer a no-account, no-server tool

It is not designed for:

  • Submitting applications automatically. The user always reviews before submitting.
  • Recruiters or hiring teams. The product is for the applicant side only.
User-role implementation notes
  • Single-user model. No accounts, no roles, no multi-tenant logic anywhere in the codebase.
  • Profile is read and written by the options page (entrypoints/options/) and consumed by the autofill engine (src/autofill/).

One-click autofill

Off by default. The user pastes their own Gemini API key, uploads a resume PDF or DOCX, and reviews extracted fields before anything is saved.

Optional AI autofill assist
For fields the rule-based pipeline cannot resolve (complex selects, radio groups), the extension can ask Gemini for a match. Off by default.

User workflow

How a user actually uses it

1
Set up the profile (once)
Open the options page and fill in the nine sections. Optionally upload a resume and let AI extract suggestions (with review). Optionally enable Google Drive backup.
2
Open a job application page
Any employer or ATS site. The content script is present but does nothing until invoked.
3
Click the toolbar icon and press Fill Form
The extension scans the page, classifies each field, and fills what it can.
4
Review colour-coded fields
Green = high confidence, yellow = medium (review), red = low (the picker overlay offers alternatives), grey = no profile data (the picker offers a deep-link to the relevant profile section).
5
Correct anything wrong, then submit
Manually picking a different profile value via the picker teaches the extension that site-specific mapping for next time. The user always submits the form themselves.
6
Undo if needed
One click clears every value the extension wrote in that session.
Data flow
  • Popup click → chrome.tabs.sendMessage → content script AUTOFILL_SCAN
  • Scan → scanner.ts finds elements → mapper.ts classifies each → store as pendingMatches (no DOM changes yet)
  • Fill → AUTOFILL_FILL → filler.ts writes values + highlighter.ts applies colour + picker.ts attaches overlay listeners
  • AI assist (if enabled) → ai.ts resolves remaining fields via Gemini
  • Deep-link to profile → content script sends OPEN_OPTIONS with focusPath → background writes session storage → options page reads it on mount or on tab focus
Data and privacy

What's stored and where

Source of truth: the privacy policy under docs/legal/privacy/.

According to the privacy policy, Job Buddy stores all profile data locally in the browser's own storage. The extension has no servers and no user accounts. Data only leaves the user's device through three opt-in features (Google Drive backup, AI resume import, AI autofill assist), and only when the user explicitly enables each one.

What's stored locally

  • Profile fields the user entered (personal info, address, salary, work history, education, languages, links, work authorization)
  • Resume or CV file, if uploaded, encoded as file data
  • Per-domain learned field mappings (signal-to-profile-path only, never the values typed)
  • Application history (records of filled forms)
  • AI settings: Gemini API key and selected model (only if AI features are enabled)
  • Drive backup state: OAuth token and bookkeeping (only if Drive backup is enabled)
  • Theme preference
According to the privacy policy: the extension does not make network requests by default, does not sell or share data, and does not use Chrome account sync. Data only reaches third-party APIs (Google Drive, Google Gemini) through opt-in features the user activates.

EEO fields

Optional gender, ethnicity, veteran status, and disability status fields are categorised as special category data under GDPR Article 9. The privacy policy states they are processed only on the basis of the user's explicit consent (which they provide by voluntarily entering values), are stored only on-device, and can be deleted from the options page at any time.

Storage technical details
  • Primary storage: chrome.storage.local (5 MB hard limit)
  • Session storage: chrome.storage.session for transient UI hints (deep-link target, dismissed prompts)
  • CV file cap: 4 MB (base64-encoded, kept under the 5 MB total quota)
  • Export bundles: exclude geminiApiKey, geminiModel, driveToken, driveBackupState
  • Drive scope: drive.appdata only — the extension cannot read the user's other Drive files
  • Relevant source files: src/utils/storage.ts, src/utils/driveSync.ts, src/utils/migrate.ts
Permissions

What the extension can do in the browser

Source of truth: wxt.config.ts.

💾
Storage
Used to save the user's profile, learned field mappings, application history, and UI preferences locally on the device. Required for the core functionality.
👤
Identity
Used only when the user opts in to Google Drive backup. Authenticates with Google via chrome.identity.launchWebAuthFlow. Not used for any other purpose.
🗂️
Active Tab
Used so the popup can send commands (Fill Form, Undo, status check) to the content script on the tab the user is currently viewing. Only the active tab is ever accessed.
🌐
Host access (*://*/*)
The content script is loaded on any web page because job application forms appear across thousands of unrelated employer and ATS domains. According to the manifest, the script is passive and does nothing until the user clicks Fill Form. When activated it reads form field metadata only (labels, attributes), not field values or page content.
🔌
Host permissions for Google APIs
generativelanguage.googleapis.com, www.googleapis.com, oauth2.googleapis.com. Used by the optional Gemini AI features and Google Drive backup.
Manifest details
  • Manifest version: 3 (MV3)
  • permissions: storage, identity, activeTab
  • host_permissions: Google API hosts only (see above)
  • Content script match pattern: *://*/*
  • Config file: wxt.config.ts
Current limitations

What it does not do today

  • Does not submit applications. The user always reviews and submits manually.
  • Does not work on Firefox, Safari, or mobile browsers. Chrome / Chromium (MV3) only.
  • CV or resume files are capped at 4 MB because of the chrome.storage.local 5 MB total quota.
  • Google Drive backup requires the user to set up their own OAuth client during development. The Chrome Web Store build uses a published OAuth client.
  • AI features require the user to obtain their own Google Gemini API key (BYOK model). The extension does not provide one.
  • Cross-device sync is only available through Google Drive backup. There is no first-party sync server.
  • Built-in browser autofill is not disabled and may overlap with the extension's behaviour on the same fields.
Implementation constraints
  • Storage cap: chrome.storage.local 5 MB hard limit (Chrome-imposed)
  • OAuth flow: Google Cloud OAuth client must be type "Web Application", not "Chrome Extension", to match chrome.identity.launchWebAuthFlow
  • Cross-context restriction: chrome.storage.session.set from content scripts is blocked unless the extension has host_permissions for the page URL — which it intentionally doesn't (only Google API hosts). All cross-context storage writes from the content script must route through the background service worker.

Technical summary

For semi-technical readers

Job Buddy is a Chrome MV3 browser extension built with the WXT framework, React 19, and TypeScript. There is no backend — the entire product runs inside the user's browser. Optional integrations (Google Drive, Google Gemini) make outbound API calls from the browser directly, with no proxy server in between.

The codebase is split into four entrypoints (background service worker, injected content script, action popup, full-page options editor) plus shared modules for the autofill engine, profile types, storage wrappers, and the optional AI integrations. Distribution is through the Chrome Web Store, with automated uploads triggered by pushing semver tags to GitHub.

Architectural boundaries
  • Background service worker (entrypoints/background.ts) — routes OPEN_OPTIONS, writes session storage for cross-context deep-links, retries pending Drive sync on browser startup
  • Content script (entrypoints/content.ts) — injected on all pages, passive until invoked, routes autofill messages
  • Popup (entrypoints/popup/) — React UI for Fill / Undo / status, plus hidden debug panel
  • Options page (entrypoints/options/) — full React profile editor (9 sections + Settings + Resume Import)
  • Autofill core (src/autofill/) — engine that runs inside the content script context
  • Resume AI (src/resume-ai/) — Gemini integration, isolated so it can fail silently without breaking core autofill

  • CI: .github/workflows/ci.yml runs type-check, lint, and tests on every push and PR
  • Setup notes: .github/SETUP.md documents Google Cloud + GitHub secret setup
